## 3. 現状・問題・選択肢・アプローチ・結果

### 3.1 メッセージフォーマット

| | |
|---|---|
| **現状** | 通信フォーマット未決定 |
| **問題** | バイナリ vs テキストのトレードオフ、将来的なパフォーマンス懸念 |
| **選択肢** | ① JSON のみ　② MessagePack のみ　③ JSON スタートで MessagePack に移行 |
| **アプローチ** | まず JSON で実装し、パフォーマンス問題が実測で出た時点で MessagePack に切り替える。基本構造は `type / sequence / timestamp / payload` で統一 |
| **結果** | **JSON ファーストで確定**。構造: `{ type, sequence, timestamp, payload }` |

---

### 3.2 認証フロー

| | |
|---|---|
| **現状** | 認証方式未決定。テスト工数とユーザー体験のバランスが課題 |
| **問題** | Apple Sign In を最初から必須にするとテスト・開発フェーズが煩雑になる。ゲストログインを先行させるとランキング・スキンの永続化が後回しになる |
| **選択肢** | ① ゲストログイン先行 → Apple Sign In 後追い　② 最初から Apple Sign In 必須 |
| **思考プロセス** | Face ID の普及でApple ID ログインの摩擦は許容範囲。ただし開発初期フェーズではゲストログインの方がテストしやすい。ランキングやスキンなど「残したいデータ」が生まれた時点で Apple Sign In を求めるのが自然なフロー |
| **結果** | **ゲストログイン先行で確定**。ランキング・スキン解放時に Apple Sign In を要求するフローに |

### 3.3 マッチング仕様

| | |
|---|---|
| **現状** | マッチングのタイムアウト・ゲームルール選択フローが未定 |
| **問題** | タイムアウト秒数をハードコードするとチューニングが困難。プライベートとパブリックで UI フローが異なる |
| **選択肢** | タイムアウト秒数: ハードコード / サーバー側変数 |
| **アプローチ** | スプラトゥーン・マリオカート・ポケモンユナイト・Overcooked を参考に画面フローを調査・設計 |
| **結果** | **タイムアウト秒数はサーバー側変数で管理**（後から調整可）。画面フローは以下に確定: |

```
プライベート / パブリック選択
  ↓
ゲームルール設定（時間・ステージ・スコア上限・チーム分け）
  ↓
マッチング待機
  ↓
試合開始
```

### 3.4 バージョン管理・アセット配信

| | |
|---|---|
| **現状** | ゲームモード追加・キャラクター追加時の配信方法が未定 |
| **問題** | Apple はアプリ内でリモートからコードをダウンロードして実行することを **禁止** している。App Store の再審査を経ずにコンテンツを追加したい |
| **選択肢** | ① すべてネイティブコードに組み込む（審査必須）　② アセット（画像・モデル・BGM 等）のみサーバー配信　③ ロジックをスクリプトとしてサーバー配信（Apple 規約違反） |
| **思考プロセス** | ロジック変更は必ず審査が必要。アセットと設定値（パラメーター）はサーバー側で管理できる。Unity には **Addressable Assets** という専用の仕組みがあり、これを使えばステージ・キャラクターモデル・画像・BGM・UI をサーバー配信できる。JSON で外部設定値も管理可能。新キャラは先にビルドに含めてサーバー側フラグで非公開にし、解放時にフラグを変えるだけで公開できる（= リーク対策として暗号化も必要） |
| **結果** | **Addressable Assets を採用**。配信物の種別を以下で整理: |

| 種別 | 配信方法 | 審査要否 |
|---|---|---|
| ゲームロジック（スクリプト） | ネイティブコードに組み込み済み | **審査必要** |
| アセット（モデル・ステージ・BGM・画像・UI・イベント） | Addressable Assets でサーバー配信 | **不要** |
| パラメーター（数値・設定値） | JSON でサーバー管理 | **不要** |
| 新コンテンツ（新キャラ等） | ビルドに同梱 → サーバーフラグで公開制御 | **不要**（初回のみ審査） |

### 3.5 クライアント↔サーバーの責務分担

| | |
|---|---|
| **現状** | どのロジックをサーバーで処理し、どれをクライアントで処理するか未決定 |
| **問題** | クライアント側で全処理すると不正が容易になる。サーバー側で全処理すると通信量・レイテンシが増大 |
| **選択肢** | ① クライアントマスター方式（1クライアントが全演算を担い、他に配信）　② サーバーオーソリティ方式（サーバーが最終判断）　③ クライアント予測 + サーバーロールバック |
| **思考プロセス** | スプラトゥーンなど実績あるゲームの設計を参考。ゴール・スコアなどの「絶対に正確でなければならない」データはサーバーに権威を持たせる。移動座標はクライアントで先行処理し、ずれがあればサーバーが訂正する方式が遅延感を抑えられる |
| **結果** | **サーバーオーソリティ + クライアント予測 + サーバーロールバック**で確定。具体的な分担: |

| 処理 | 担当 |
|---|---|
| ゴール・スコア判定（最終） | **サーバー** |
| ログイン・ランキング | **サーバー**（REST） |
| プレイヤー座標（先行処理） | **クライアント**（Client-side Prediction） |
| 座標のズレ修正 | **サーバーロールバック** |
| プレイヤー座標の他クライアントへの伝達 | **サーバー中継**（将来 WebRTC に移行検討） |

> **座標補間（座標の滑らかな補間処理）** はクライアント側で実装予定（前回から継続課題）

### 3.6 リプレイ機能の設計議論

| | |
|---|---|
| **現状** | 将来的なリプレイ機能を検討中 |
| **問題** | 動画として保存するとデータ量が大きく、視点が1つに固定される。ゲームパラメーター変更後に古いリプレイを再生すると挙動が変わる |
| **選択肢** | ① 動画録画　② 入力ログ + パラメーター記録方式（スプラ方式） |
| **思考プロセス** | スプラトゥーンはリプレイを動画ではなく「入力ログ + パラメーター集合体」として保存している。これにより複数視点（敵視点・俯瞰視点）での再生が可能。ただしパラメーターを変更すると過去リプレイの挙動が変わる問題があり、対応するにはバージョンごとにゲームエンジン状態を保持し続ける必要がある（非現実的）。現時点では指サッカーにリプレイ機能は **不要** |
| **結果** | **リプレイ機能は v1 スコープ外**。将来検討する場合は入力ログ方式が有力 |

---

### 3.7 ゲームモード・チーム構成

| | |
|---|---|
| **現状** | 対戦人数・チーム構成が未定 |
| **問題** | 3チーム以上の対応はサーバー設計・クライアント設計双方で複雑度が増す |
| **選択肢** | ① 2チームのみ　② 3チーム以上対応 |
| **思考プロセス** | スプラトゥーンも2陣営のみ。3チーム以上にするとゴール判定・スコア集計ロジックが複雑になる。オウンゴールが成立しなくなる等の問題もある。YAGNI の観点から、今回必要でない拡張性を持たせるのは工数の無駄 |
| **結果** | **v1 は2チーム固定**。モード構成: |

| モード | 内容 |
|---|---|
| ソロ練習 | 1人用トレーニングモード |
| 1v1 | 2名対戦 |
| 2v2 | 4名対戦 |
| 3v3 | 6名対戦 |

> ビリヤード・ストラックアウト等の他ゲームモードは将来検討（v1 スコープ外）

---

### 3.8 パブリックマッチング

| | |
|---|---|
| **現状** | パブリックマッチング（見知らぬ人とのマッチング）を実装するか未定 |
| **問題** | リリース直後にマーケティングなしではプレイヤーが集まらない。マッチング待機中に誰も来なければ UX が悪化する（「1人もいない」状態での待機は期待値を裏切る） |
| **選択肢** | ① パブリックあり　② パブリックなし（プライベートのみ） |
| **思考プロセス** | パブリックを入れる主な理由は「一人でも見知らぬ人と遊べる」だが、集客（マーケティング）が伴わないと機能しない。技育展での展示は想定されているが、継続的なプレイヤーベースがない段階ではパブリックの価値が低い。悪い UX を出すくらいならない方がよい |
| **結果** | **v0 はパブリックマッチングなし**（プライベートマッチのみ）。将来的にマーケティング施策と合わせて検討 |
