## 3. 現状・問題・選択肢・アプローチ・結果

### 3.1 MediaPipe × Unity パフォーマンス（カシュー）

| | |
|---|---|
| **現状** | iPhone 実機ビルド成功。モバイルで MediaPipe は動作確認済み。Profiler 計測で Total Allocated 5.48GB、うち Native 4.06GB |
| **問題** | Native メモリの大半が MediaPipe 推論エンジン由来と推定。ゲーム本体に組み込むと常時 4.4GB を消費する。TextMeshPro によりテクスチャ数が 1031 枚（本来 30 枚程度）に膨張、RenderTexture も重い |
| **選択肢** | ① 不要な MediaPipe パッケージ（手以外のモデル）を削除　② ランドマーク描画の最適化　③ MediaPipe の起動を試合中のみに限定 |
| **アプローチ** | MediaPipe Unity パッケージに手以外のモデル（ボーン・ボディ等）が含まれているか確認し、不要なものを削除。TextMeshPro → 標準テキストへの置き換えで RenderTexture 削減。③ は暫定対処として採用 |
| **結果** | 技術的には使用可能と確認。パフォーマンス最適化は継続課題。次の月曜までに改善可能な箇所に着手予定 |

### 3.2 サーバー設計・バージョンアップ戦略（りーせ）

| | |
|---|---|
| **現状** | Ver.2.0 のコードを読み、設計課題（テストしにくい構造・権限判定の問題・サッカー前提のハードコーディング）を確認 |
| **問題** | 共通基盤（マッチング・認証・タイムアップ処理）が再利用できない設計になっている。毎回フルビルド・審査が必要なリリースフローはコストが高い |
| **選択肢** | ① スキン・ステージ・アイテム・BGM・UI 等のアセットはサーバー配信で管理し審査不要に　② ゲームシステム追加時のみクライアント更新・再審査 |
| **アプローチ** | プロトタイプ段階はクライアント・サーバー分離のシンプルな構成を維持。リリース時点でサーバー配信方式に切り替える |
| **結果** | 方針として合意。詳細実装はリリースフェーズで決定 |

### 3.3 通信プロトコル選定（りーせ・ざきこう）

| | |
|---|---|
| **現状** | りーせが REST / WebSocket / WebRTC を調査。ざきこうが gRPC / WebSocket / MessagePack を調査 |
| **問題** | リアルタイム性が必要な処理（マッチング・同期・スコア更新）と不要な処理（ログイン・プロフィール更新）が混在している |
| **選択肢** | ① REST + WebSocket　② gRPC + WebSocket　③ WebRTC（P2P） |
| **アプローチ** | gRPC はブラウザからのアクセスにサーバー中継が必要でコストが高い。WebRTC はサーバーとの相性が悪い。REST（非リアルタイム処理）+ WebSocket（リアルタイム処理）の組み合わせが双方の調査で一致 |
| **結果** | **REST + WebSocket を採用方針として決定**（ざきこうとの詳細確認後に正式決定予定）。データフォーマットは JSON から開始し、負荷が問題になれば MessagePack へ移行 |

### 3.4 データベース構成（りーせ）

| | |
|---|---|
| **現状** | DB 構成が未定 |
| **選択肢** | ① PostgreSQL のみ　② PostgreSQL + Redis　③ PostgreSQL + Redis + MongoDB |
| **アプローチ** | 永続データ（プレイヤー情報・レート・スコア履歴）は PostgreSQL、試合中の揮発データ（ルーム状態・マッチング・セッション・試合スコア）は Redis で管理。MongoDB（プレイ履歴・実績）はリリース時に追加 |
| **結果** | **プロトタイプ: PostgreSQL + Redis** を採用方針として決定。MongoDB はリリース後に移行 |

|---|---|

### 3.6 セキュリティ・既存コードの問題（ざきこう）

| | |
|---|---|
| **現状** | App ID（Dart の設定ファイル）がコミット済み。キック判定がマスター以外のクライアントからも `AddForce` を呼び出せる実装になっている |
| **問題** | App ID 漏洩はリリース前に必ず修正が必要。非マスタークライアントからの `AddForce` 呼び出しによりボールが瞬間移動するバグが発生し得る |
| **アプローチ** | App ID は環境変数または秘匿設定ファイルで管理（`.gitignore` 追加）。ゴール・ボール移動処理はマスタークライアントに一元化 |
| **結果** | 未対応。次回以降で修正対応 |

### 3.7 接続・マッチングのエラーハンドリング（ざきこう）

| **アプローチ** | 通信プロトコル選定が固まった後に合わせて設計 |
| **結果** | 未対応。サーバー設計フェーズで対処 |

### 3.8 デザイン・世界観（なかむー）

|---|---|
| **現状** | 就活ポートフォリオ制作の一環でゆびサッカーの操作写真（プレイ中の手の写真）を撮影済み。UI 参考としてフォールガイズ・ゼンゼロを調査 |
