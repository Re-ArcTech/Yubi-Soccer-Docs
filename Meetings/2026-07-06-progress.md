### 3.2 カシューの進捗

| | |
|---|---|
| **状況** | Claude Fable が1週間解禁されたタイミングで MediaPipe モーションキャプチャーパッケージのリファクタリングに注力 |

**MediaPipe パッケージ改善内容**

| 改善項目 | 内容 |
|---|---|
| **名前空間の整理** | パッケージ化に伴うインポート先との競合を解消 |
| **API のクリーンアップ** | ごちゃついていた API を整理・再設計 |
| **型判別の見直し** | 列挙型（enum）による手の形（ピース・キック等）の判別精度改善 |
| **デバッグ API の強化** | カメラ出力・指の本数・ジェスチャー表示などを確認できる UI を強化 |
| **閾値の UI 変更** | キック・走るなどのジェスチャー判定閾値を実行時に UI から変更可能に |
| **エラー通知の追加** | 「指が画面外に出ています」等のエラーをイベント発火式でテキスト表示 |
| **コードパフォーマンス改善** | 不要な処理を削減 |
| **録画機能の追加** | カメラなしでテンプレートとなる指の動きを録画・インポート・再生できる仕組みを実装。これにより実機カメラなしでの動作確認が可能になった |
| **変数可視化の強化** | より多くのデバッグ変数を画面上に表示可能に |
| **パッケージ構成の見直し** | 技術検証時の暫定構成を整理・再設計 |

> **背景**: 初期のパッケージ化は技術検証目的だったため API 設計が荒かった。本格的な統合前のこのタイミングで整理し、以降の開発コストを下げる判断をした。

**次週の予定**

Unity 上でのアタッチ・オブジェクト配置を行い、ソロ練習モード（1人用）を実際に動かせる状態にする。

**目標**: **2週間以内に1v1 の対戦動作を確認**

### 3.5 今後の技術課題（カシュー → りーせへ依頼）

繋ぎ合わせを進める中で重要になってくる課題として、りーせに先行検討を依頼。

| 課題 | 内容 |
|---|---|
| **タイミング同期** | ラグを前提としたとき、全クライアントのスタートラインを揃える処理が各所で必要。マッチング完了・キックオフ・ゴール後のスコア確定など。各タイミングでの同期設計を考えておいてほしい |
| **データ整合性** | 試合後にクライアント側でセーブデータを作成してサーバーに渡す仕組み。試合中断・プレイヤー切断時の処理など |
| **切断への対応** | スプラトゥーンのようにプレイヤーが途中で落ちた場合の扱い（試合継続 or 中断、データの保全等） |
| **状態の非同期処理** | 現状、試合状態の保存処理で試合が一時停止する問題がある。DB 書き込みを非同期化して試合進行をブロックしないよう対応が必要 |
