**① サーバー基盤**

| | |
|---|---|
| **実装内容** | 接続 → バージョンチェック → ログインという基本フローを実装 |
| **設計方針** | 認証・DB・通信方式など未確定部分をインターフェースで切り出し、後から差し替えられる構造にした。土台を崩さずに機能追加できる |

**② プレイヤー同期**

| | |
|---|---|
| **実装内容** | 各プレイヤーが自分の座標を**約 20Hz** でサーバーに送信。サーバーが他の全プレイヤーに中継 |
| **追加機能** | チーム振り分け（赤/青）。途中参加時に現在の全プレイヤー座標を送信 |
| **重要仕様** | **時刻はサーバー基準で統一**。クライアントは端末の時計を使わないこと |

**③ ボール同期**

| | |
|---|---|
| **実装内容** | ボールの物理計算をサーバー側で **60Hz** 実行。結果をクライアントへ配信 |
| **実装済み** | XYZ 座標・重力・バウンド・キック方向・キックチャージ |
| **思考プロセス** | クライアントはボール操作のロジックを持たず、「キックしたい（方向 + チャージ量）」というリクエストをサーバーに送るだけにしたい。物理演算はすべてサーバーが担う |

**④ ゴール判定**

| | |
|---|---|
| **実装内容** | サーバーがボール位置とゴールエリアを照合し、整合性が取れたら得点を確定 |
| **クライアントの役割** | クライアント側のゴールエリア判定は**演出（アニメーション・エフェクト）用のみ**。即時加点はしない |
| **結果** | 不正・通信ズレによるゴール誤認を防ぐサーバーオーソリティ構成が完成 |

### 3.3 技術雑談（スプラトゥーン・Apex の実装考察）

定例終了後、スプラトゥーンやApexの実装について議論した。決定事項ではないが、今後の設計参考として記録する。

**マッチング完了後の待機時間（スプラトゥーン）**

| 仮説 | 内容 |
|---|---|
| 全クライアントの ACK 待ち | 8人全員が「マッチング確認OK」を返すまでサーバーが待機している可能性 |
| 専用セッション/部屋の確立 | 8人専用の通信セッションをサーバー側で確保する処理に時間がかかる可能性 |
| シーン読み込み待ち | 全員のロードが揃うまで待機してから試合開始する |

→ **指サッカーへの示唆**: マッチング完了→試合開始の間に、全員のACK確認・セッション確立・シーン同期の処理を設計に組み込む必要がある。

**ステージ塗り状態の同期（スプラトゥーン）**

| 仮説 | 内容 |
|---|---|
| 入力ベースの演算 | 各プレイヤーの「塗るアクション」イベントを起点に、サーバーが面積計算する |
| ピクセル状態の同期ではない | 全マップのピクセル状態を同期するのは現実的でなく、イベント+演算で再現している可能性が高い |

**試合終了後のデータ保存（セーブ整合性）**

- 試合中はカレントデータをキャッシュ保持
- 試合が正常終了したタイミングでDBへ書き込み
- 異常終了に備えた整合性チェックが必要

→ **指サッカーへの示唆**: 試合終了後の処理設計（スコア確定 → DB書き込み → リザルト表示）を今後整理する必要がある。
