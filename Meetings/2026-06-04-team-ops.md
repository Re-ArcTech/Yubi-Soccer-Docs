## 3. 現状・問題・選択肢・アプローチ・結果

### 3.1 GitHub メンバー招待

| | |
|---|---|
| **現状** | オーガナイザーへの招待はしていたが、Docs リポジトリへの招待が未実施 |
| **問題** | ざきこうがリポジトリにアクセスできていない |
| **アプローチ** | 全員をメンバーとして招待（カシュー・なかむー・りーせ・ざきこう）。権限はメンバーで統一、問題があればオーナーに変更 |
| **結果** | 招待実施。引き続き問題あれば権限変更する |

### 3.2 ターゲットプラットフォーム

| | |
|---|---|
| **現状** | チームメンバー全員が iPhone ユーザー。日本国内の iPhone シェアは約 48% |
| **問題** | Android 対応を行うにはビルド・テスト環境が必要で工数がかかる |
| **選択肢** | ① iOS のみ　② iOS 優先・後から Android 追加　③ 最初からクロスプラットフォーム |
| **アプローチ** | まず iOS に集中し、余力があれば Android 対応（技育展近くのタイミング等） |
| **結果** | **② を採用決定**。当面は iOS のみ |

### 3.3 ビルド・デプロイ戦略

| | |
|---|---|
| **現状** | 毎回 Mac でフルビルドが必要。なかむーが Mac を持っていないため都度集まる必要がある |
| **問題** | デプロイコストが高く、開発サイクルが回しにくい |
| **選択肢** | ① 毎回フルビルド　② 初回ビルド後はアップデートデータだけ配信（近代ゲーム方式） |
| **アプローチ** | ② の方向で検討。Sandbox 等の仕組みも調査中 |
| **結果** | 未決。りーせ・ざきこうが通信・サーバー調査と並行して方式を検討する |

### 3.4 通信・サーバー方針

| | |
|---|---|
| **現状** | まだ何も決まっていない。Photon 使用の可否、データベース構成、サーバー側の実装方式が未定 |
| **問題** | 方針が固まらないと実装が進められない |
| **選択肢** | Photon / 独自サーバー / Sandbox など複数案あり |
| **アプローチ** | ざきこう・りーせがもう少し下調べをしてから詳細を詰める |
| **結果** | 未決。次回以降でサーバーサイドミーティング（ざきこう・りーせ）を設けて決定する |

### 3.5 キャラクターパラメーター管理

| | |
|---|---|
| **現状** | どこでパラメーターを管理するか未定 |
| **選択肢** | ① スプレッドシート　② データベース　③ ゲーム内アセット |
| **アプローチ** | 通信・サーバー方針が固まってから決定 |
| **結果** | 未決 |

### 3.6 MediaPipe × Unity

| | |
|---|---|
| **現状** | MediaPipe Unity パッケージが存在することを確認。専用プロジェクトで試行中 |
| **アプローチ** | 残り時間でパッケージを試す |
| **結果** | 引き続き検証中 |

### 3.7 技術アウトプット（Qiita 記事）

| | |
|---|---|
| **現状** | 各自の技術的な進捗が文字として残っていない |
| **問題** | 後から振り返れない・他メンバーに共有できない・LT 等のアウトプットに活かせない |
| **選択肢** | ① Qiita 記事として書く　② ディスコード報告のみ　③ ドキュメントリポに残す |
| **アプローチ** | **日報習慣 → モジュール完成ごとに Qiita 記事**。ディスコードで進捗報告を続けつつ、1 モジュール終わったら 1 記事を目標にする（最初は 1 時間、慣れたら 40 分程度） |
| **結果** | 採用。強制ではないが積極的に取り組む。Docs リポは議事録用とし、技術アウトプットは Qiita で行う |

## 4. 決定事項

| # | 決定内容 | 担当 | 期限 |
