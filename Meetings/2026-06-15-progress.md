## 3. 現状・問題・選択肢・アプローチ・結果

### 3.1 デザイン進捗（なかむー）

| | |

| **現状** | 就活用ポートフォリオ（PDF）が完成。ゆびサッカーの操作写真を撮影・掲載済み。UI 参考としてフォールガイズ × ゼンゼロ方向を調査中 |

| **結果** | 9月以降に本格参加。それまでは調査・インプット期間とする |

### 3.2 MediaPipe パフォーマンス最適化（カシュー）

| | |
|---|---|
| **現状** | 前回計測でメモリ使用量 5.48GB（Native 4.06GB）と過大だった |
| **問題** | このままゲームに組み込むと常時 4.4GB を消費し実用不可 |
| **選択肢** | ① TextMeshPro → Legacy テキストへの置き換え　② MediaPipe 推論グラフの描画テクスチャ最適化　③ AI 生成コードの不要オブジェクト削減 |
| **アプローチ** | ① ② ③ をすべて実施。不要モデルは元々インストールされていなかったことを確認 |
| **結果** | **メモリ使用量を 356MB まで削減（約 90% 削減）**。動作も問題なし。主因は TextMeshPro による RenderTexture・テクスチャ数の膨張だった。ゲームへの組み込みが現実的なレベルになった |

### 3.3 通信ベンチマーク結果（りーせ）

| | |
|---|---|
| **現状** | REST / WebSocket / WebRTC の 3 方式を実際にベンチマーク計測 |
| **問題** | ホスティングリージョンによるレイテンシ差、パケットロス時の挙動差を把握できていなかった |
| **選択肢** | ローカル / シンガポール（Render）/ 東京（AWS）の 3 環境で計測 |
| **アプローチ** | 各環境で往復遅延を計測。パケットロス 8% を意図的に発生させて WebSocket / WebRTC の挙動差を確認 |
| **結果** | シンガポール（Render）約 100ms に対し東京（AWS）約 15ms と **6〜7 倍の差**。パケットロス発生時、WebSocket は順序保証のため固まるリスクがあるが WebRTC は損失を許容しフリーズしない。1000 人以上の同時接続に耐えることも確認 |

### 3.4 通信方式の使い分け方針（りーせ・ざきこう）

| | |
|---|---|
| **現状** | 前回の方針（REST + WebSocket）をベンチマーク結果で検証 |
| **問題** | プレイヤー座標などロス許容可能なデータと、ゴール・スコアなど損失不可なデータで最適な方式が異なる |
| **選択肢** | ① WebSocket のみ　② WebSocket + 将来的に WebRTC を併用 |
| **アプローチ** | ゴール・スコア情報（損失不可）は WebSocket。ログイン・ランキングは REST。プレイヤー座標（最新値のみ必要）は将来的に WebRTC へ移行 |
| **結果** | **WebSocket メイン・REST 併用で確定**。プレイヤー座標の WebRTC 化は将来対応として保留。ホスティングは **AWS 東京リージョンを優先** |
